# 暂停任务已入库候选展示设计

## 目标

采集任务运行中被用户暂停后，候选池立即展示暂停前已经真正写入红人库的红人。任务自然完成时继续使用现有完整收尾逻辑，并默认展示已入库红人。

## 数据流

采集运行器每完成一个持久化批次，已经能够获得本批次的写库结果和对应红人 ID。批次提交时同步重建当前任务的“已入库”候选快照，候选池因此不再依赖任务结束后的统一收尾。暂停只改变任务状态；候选数据在最近一次批次提交时已经落库。

## 行为规则

- 运行中：每个持久化批次提交后更新已入库候选快照。
- 暂停：保留最近一次已提交批次的候选快照，只包含 `inserted` 状态。
- 恢复：继续按批次更新并按任务内身份去重。
- 正常完成：现有最终候选明细生成逻辑继续执行，可保留过滤、失败等诊断记录；页面默认筛选已入库。
- 暂停发生在外部 API 请求中：等待该请求返回及当前小批次提交，不展示未完整写库的数据。

## 一致性与错误处理

候选快照与任务进度在同一数据库事务中提交。快照生成失败时不伪造已入库数量，运行器记录错误并沿用现有任务失败处理。候选快照按任务整体替换，避免恢复或重试造成重复记录。

## 测试

- 批次提交时只写入 `inserted` 候选。
- 暂停收尾不清空已有候选快照。
- 正常完成仍写入完整候选明细。
- 候选数量与任务已入库统计一致。
