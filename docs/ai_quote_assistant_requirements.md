# 自动报价助手智能化需求文档

## 1. 背景

当前系统已经具备报价引擎、表格解析、价格库匹配、报价解释、局部试算和轻量问答能力，但整体体验仍偏“工具执行器”：用户一上传文件或问一句话，系统容易按固定流程响应，缺少像业务助理一样先判断意图、再给合适答案的能力。

本需求文档目标是把系统从“自动报价工具”升级为“报价客服/业务核价助手”：

- 能报价，也能解释为什么这么算。
- 能识别用户是要问问题、看差异、改参数、查材料，还是正式出报价。
- 不盲目自动报价，优先尊重用户文字需求。
- 对不确定信息先追问，减少误算和误导。

## 2. 当前能力诊断

### 2.1 已具备能力

| 能力 | 当前实现 | 说明 |
|---|---|---|
| 上传表格报价 | `server.py` + `sheet_parser.py` + `quote_engine.py` | 可解析 BOM/需求表并生成明细和三档报价 |
| 意图路由 | `request_intent_router.py` | 已有 quote / quote_patch / explain / qa / clarify 分流 |
| 结构确认 | `build_structure_confirmation_payload` | 上传后可先让用户确认结构、用量、单价 |
| 报价解释 | `quote_explain.py` | 可基于上一次报价解释材料、加工费、杂费、开模、毛利 |
| 局部改价/试算 | `quote_agent` + `local_quote_patch.py` | 可处理箱子改价、数量改价、局部参数试算 |
| 价格库问答 | `qa_rag.py` | 可查价格库、历史报价、README 文档 |
| 澄清问题 | `clarify_once.py` | 可在缺少上下文或改价目标不清时追问 |

### 2.2 当前主要问题

| 问题 | 表现 | 影响 |
|---|---|---|
| 意图判断仍偏规则化 | 用户问“为什么不一样”，系统可能回复生硬或要求重新报价 | 用户觉得“不聪明”“答非所问” |
| 答疑和报价结果割裂 | 有报价卡，但解释不一定能围绕用户截图/业务员结果回答 | 难以完成真实核价沟通 |
| 上传附件语义不够细 | 有文件时容易被理解为“要报价”，即便用户想解释差异 | 误触发报价流程 |
| 对比能力弱 | 无法稳定识别“图一业务员算的、图二系统算的”的关系 | 用户追问价差时回答不够落地 |
| 回复话术偏系统化 | “当前没有进行中的报价”等提示较硬 | 不像客服，不利于业务使用 |
| 缺少任务状态感知 | 不清楚当前是在报价、解释、改价、确认结构、生成报价单 | 多轮对话容易跑偏 |
| 缺少可视化解释 | 价差原因主要文字说明，缺少表格化对比 | 客户难以快速看懂 |

## 3. 产品目标

### 3.1 一句话目标

把系统升级为“懂报价上下文、能主动澄清、能解释差异、能局部试算、能输出报价”的智能报价客服。

### 3.2 核心原则

1. 用户文字优先：附件只是资料，不能覆盖用户文字意图。
2. 不确定先问：缺少报价目标、对比对象、价格口径时先澄清。
3. 解释先给结论：先说差异来自哪里，再展开公式。
4. 数字可追溯：每个金额都能回到材料、用量、单价、加工费、杂费、开模或毛利。
5. 试算不覆盖主报价：局部改价默认生成试算，用户确认后才升级主报价。

## 4. 目标用户场景

### 4.1 正式报价

用户输入：

> 帮我按这个表报价

系统行为：

1. 解析附件。
2. 展示结构/明细预览。
3. 用户确认后生成正式报价。
4. 输出材料合计、成本价、EXW/FOB、三档报价。

验收标准：

- 明确报价意图才进入正式报价。
- 上传后默认先确认结构，除非用户明确“直接报价”。
- 缺关键字段时给出具体缺什么，而不是泛泛说失败。

### 4.2 解释怎么算

用户输入：

> 你怎么算的？

系统行为：

1. 引用当前报价。
2. 给出一句结论。
3. 分解成本：
   材料 + 加工费 + 杂费/管理费 + 开模均摊 = 毛利前成本，再反推 EXW/FOB。
4. 列出主要金额来源。

验收标准：

- 不重新报价。
- 不改价格库。
- 回复中必须包含成本公式和关键金额。

### 4.3 解释为什么和业务员不一样

用户输入：

> 第一个计算表格第二个是业务员算的，第三个是你算的，为啥不一样？

系统行为：

1. 判断这是“差异解释”，不是新报价。
2. 先问清或自动识别：哪个图/表是业务员结果，哪个是系统结果。
3. 给出差异摘要：
   - 总价差多少
   - 最大差异项是哪几项
   - 是用量差、单价差、加工费差、箱子费差，还是毛利口径差
4. 输出对账表。

验收标准：

- 有附件也不能直接报价。
- 如果无法确定图一/图二/图三关系，先追问。
- 回复必须围绕“为什么不一样”，不能只给报价结果。

### 4.4 局部改价试算

用户输入：

> 箱子按 6 元一个，加工费按 8.5，成本是多少？

系统行为：

1. 基于当前报价做试算。
2. 修改箱子费和加工费。
3. 输出新成本、原成本、差额。
4. 标注“这是试算，未覆盖主报价”。

验收标准：

- 默认不覆盖当前主报价。
- 用户说“以这个为准”后才保存为主报价。
- 回复必须同时给原值、新值、差额。

### 4.5 材料/工艺问答

用户输入：

> 600D 塔丝隆是什么材料？

系统行为：

1. 不进入报价流程。
2. 查询价格库、历史报价或本地文档。
3. 给出材料解释、常见用途、价格参考。

验收标准：

- 不要求用户上传表格。
- 不生成报价卡。
- 若价格库无结果，说明“暂未收录”，并建议补价方式。

### 4.6 功能询问

用户输入：

> 你还有哪些功能？

系统行为：

输出简洁功能列表：

- 上传 BOM/需求表生成报价
- 解释报价怎么算
- 对比业务员报价差异
- 改数量/改单价/改加工费试算
- 查材料价格和历史报价
- 生成报价单 PDF

验收标准：

- 不要求用户“具体说要改哪一项”。
- 不进入报价流程。

## 5. 功能需求

### FR-01 意图识别升级

系统需在每次请求开始时识别用户意图：

| 意图 | 示例 | 目标分支 |
|---|---|---|
| 正式报价 | 帮我报价、按这个表报价 | quote |
| 结构确认 | 先看看表格识别对不对 | clarify / structure_confirmation |
| 报价解释 | 怎么算的、为什么这么贵 | explain |
| 差异对比 | 为什么业务员算的和你不一样 | compare_explain |
| 局部试算 | 箱子改 6 元、数量改 300 | quote_patch |
| 材料问答 | 600D 是什么、YKK 多少钱 | qa |
| 功能说明 | 你能做什么 | capability_help |
| 后台操作 | 价格库怎么更新 | admin_help |

优先级：P0

验收标准：

- 有附件时也必须先看文字意图。
- 解释类问题不得触发 `calculate_quote`。
- 功能询问不得被识别为改价。

### FR-02 对比解释模式

新增“对比解释”能力，用于处理业务员报价、系统报价、截图/表格之间的差异。

输入来源：

- 当前系统报价结果
- 用户上传的业务员截图或表格
- 用户文字里提到的金额，例如 69.2、59.77、35.25

输出内容：

| 模块 | 内容 |
|---|---|
| 结论 | 差异主要来自哪 1-3 项 |
| 总差异 | 业务员金额、系统金额、差额 |
| 明细对比 | 材料、加工费、箱子费、杂费、开模、毛利 |
| 建议 | 先核对哪个字段 |

优先级：P0

验收标准：

- 支持“图一/图二/图三”引用。
- 无法识别截图金额时提示用户标注金额。
- 不直接重跑报价。

### FR-03 解释回复话术优化

报价解释需采用业务客服式结构：

```text
结论：这次差异主要在 xxx，不是单纯算错。
拆解：材料 xx + 加工 xx + 箱子 xx + 杂费 xx + 开模 xx = 成本 xx。
差异点：业务员可能用了 xx 口径，系统用了 xx 口径。
建议：优先核对 xxx。
```

优先级：P0

验收标准：

- 禁止只输出一长段公式。
- 必须先给结论，再给明细。
- 对用户质疑要直接回应，不绕到“请上传表格”。

### FR-04 多轮上下文管理

系统需要明确当前会话状态：

| 状态 | 说明 |
|---|---|
| no_quote | 没有当前报价 |
| structure_pending | 已解析表格，待确认结构 |
| quote_ready | 已有主报价 |
| trial_ready | 有试算结果，未覆盖主报价 |
| explain_mode | 正在解释当前报价 |
| compare_mode | 正在对比两份结果 |

优先级：P1

验收标准：

- 用户追问“为什么”时能引用上一单。
- 用户说“按这个为准”时能把试算升级为主报价。
- 用户上传新文件时不丢失当前对比上下文，除非明确开启新报价。

### FR-05 附件语义识别

上传文件/图片后，系统需要判断附件用途：

| 附件用途 | 示例 | 行为 |
|---|---|---|
| 新报价资料 | BOM、需求表 | 解析并等待确认 |
| 业务员报价截图 | 手写/Excel 结果截图 | 提取金额，用于对比 |
| 系统报价截图 | 用户截当前系统结果 | 识别为解释上下文 |
| 材料图片 | 包款图片、结构图片 | 辅助尺寸/工艺判断 |

优先级：P1

验收标准：

- 上传图片不默认等于报价。
- 多附件时可让用户标注“哪个是业务员，哪个是系统”。
- 识别失败时用一句话追问，不进入长流程。

### FR-06 可视化对账表

解释差异时输出对账表：

| 项目 | 系统 | 业务员/对方 | 差异 | 可能原因 |
|---|---:|---:|---:|---|
| 材料 | 21.75 | 22.50 | +0.75 | 面料用量/单价不同 |
| 加工费 | 8.50 | 8.50 | 0 | 一致 |
| 箱子费 | 6.00 | 6.00 | 0 | 一致 |
| 杂费 | 4.00 | 3.00 | -1.00 | 管理费口径不同 |

优先级：P1

验收标准：

- 至少列出 4 类成本项。
- 差异最大的项排前。
- 每一项给出可核对建议。

### FR-07 功能说明问答

新增能力介绍入口。

用户问：

> 你会什么？
> 你有哪些功能？
> 这个系统能干嘛？

系统应回答：

- 我能做什么
- 你可以怎么问
- 哪些情况需要上传表格
- 哪些情况可以直接文字追问

优先级：P1

验收标准：

- 不误判为报价或改价。
- 回复控制在 8 行内。

### FR-08 追问策略优化

系统只在必要时追问，追问必须具体。

不好的追问：

> 请再具体说一下。

好的追问：

> 你是想对比业务员报价和系统报价，还是想让我重新按这份表报价？

优先级：P1

验收标准：

- 每次追问最多给 2-3 个选项。
- 选项必须能直接推进下一步。

## 6. 非功能需求

### NFR-01 不误算

解释、问答、功能介绍不得触发报价计算。

### NFR-02 可追溯

报价结果和解释必须保留来源：

- 来自表格
- 来自价格库
- 来自系统估算
- 来自用户手动指定

### NFR-03 低延迟

意图识别优先使用本地规则，LLM 只做补充。

目标：

- 本地路由小于 200ms
- 普通解释小于 1s
- LLM 解释可异步增强

### NFR-04 稳定兜底

LLM 不可用时仍能用本地规则回复：

- 成本拆解
- 改价试算
- 价格库查询
- 澄清问题

## 8. 优先级路线图

### P0 先解决“死板”和“误触发报价”

1. 完善上传 + 文字意图优先路由。
2. 新增功能说明 intent。
3. 新增对比解释 intent。
4. 解释模式禁止自动报价。
5. 解释回复改成“结论 + 拆解 + 建议”。

### P1 提升业务可用性

1. 建对账表组件。
2. 支持图一/图二/图三附件标注。
3. 增强局部试算后确认主报价。
4. 优化澄清问题模板。
5. 增加会话状态展示。

### P2 做成长期智能助手

1. 业务员报价截图 OCR/视觉提取。
2. 历史报价自动检索相似案例。
3. 自动学习价差原因。
4. 报价异常风险评分。
5. 后台可配置报价规则。
